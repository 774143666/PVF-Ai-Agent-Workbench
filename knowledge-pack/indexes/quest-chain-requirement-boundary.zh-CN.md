## 等级、职业、成长限制

| 字段 | 主目标静态结论 | 边界 |
| --- | --- | --- |
| `[level]` | 注册任务几乎都有等级区间；常见 `99 99` 代表称号/训练/特殊桶的静态区间写法。 | 不证明服务端最终等级检查只看该字段。 |

| `[target character]` | 在部分任务中出现职业字符串 + 数字列。 | 只证明任务文件写了多职业目标块。 |

## 辅助对照提示

辅助对照 PVF 的注册任务更多，并出现主目标注册链路没有的 type，例如 `[mobile]`、`[get item]`、`[send chatting]`、`[use fortune coin]`。这些只能提示版本差异；主目标 Workbench 不把它们提升为主目标注册事实。

## 静态与动态边界

静态只读可以确认：

- 哪些任务 ID 注册到 `.qst`。
- 哪些字段、type/subtype、条件列形在主目标注册任务中存在。
- 哪些 ID 按正确 registry 能闭合。
- 哪些前置 ID、未注册文件和辅助差异需要风险提示。

静态只读不能确认：

- 任务在实机可接取、可完成、可放弃或可自动接续。
- 计数器实际增加，例如被击、背击、破招、连击、群体攻击、消耗品次数、分解次数。
- `[monster reward item]` 的掉率、任务物品掉落、组队共享和服务端放行。
- 固定/可选奖励发放成功、背包空间检查、称号 UI、PVP 最终规则。
- 客户端文本、图标、称号、NPC、地图资源完整。

## 下一步实机测试建议

最小实机测试不应从改 PVF 开始。建议只选已注册、低风险、能快速触发的代表任务：

1. `[condition under clear]` subtype 8/9/10：分别测试背击、破招、最高连击计数是否按条件文本增长。
2. `[seeking] + [monster reward item]`：接任务前后对比同一怪物是否出现任务物品候选掉落。
3. `[meet npc]`：验证接取 NPC、完成 NPC、`-1` 完成入口的 UI 行为。
4. `[use item]` 或 `[disjoint item]`：验证称号簿累计计数是否增长。
5. 对未注册 `[mobile]` / `[pvp match]` 文件只做存在性记录，不做可接取测试，除非先找到显式调用入口。
